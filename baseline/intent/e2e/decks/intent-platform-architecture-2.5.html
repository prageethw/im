Why We're Building This

Outcome-Driven Service
Instead of Low-Level Commands

The Problem Today

Network operators manually translate service goals into device-level configs — error-prone, slow, and hard to govern at scale across Telstra's 22M+ customer services.

The Opportunity

Express desired outcomes — e.g. "99.99% availability for selected production network slices" — and let the platform interpret, resolve, apply, and assure automatically.

Strategic Alignment

Aligns with TMF 921 Intent Management v5.0 standard, enabling interoperability with industry partners, OSS and BSS systems, and future AI control loops.

First Use Case

Critical regulated enterprise connectivity — selected production network slices — where uptime guarantees, latency SLAs, and automated remediation are non-negotiable.

Two Distinct Phases

Definition-Time vs Runtime

📋
Definition-Time
Governed by ID MS, the Librarian
1.Author DRAFT IntentSpecification
2.Define expression contract, schema, targets, constraints, preferences
3.Validate governance, policy, ownership, and compatibility
4.Activate DRAFT → ACTIVE official version
5.New version in the same spec family becomes ACTIVE; previous ACTIVE version becomes RETIRED
6.Publish lifecycle events, refresh active-spec cache, and allow only ACTIVE IntentSpecifications for new runtime Intent admission
⚡
Runtime
Coordinated by Controller, Brain, Dispatcher and Guardian; executed by external network systems
1.Consumer submits new runtime Intent to IC MS
2.IC MS, the Controller, validates against ACTIVE IntentSpecification and admits request
3.II MS, the Brain, resolves semantics and coordinates required decision systems
4.Brain hands service-ready configuration to Guardian (IA MS) so approved configuration can be applied and monitoring can start
5.External execution layer applies configuration; ICB MS relays callbacks
6.Guardian evaluates callbacks and observations, then emits IntentAssuranceEvent; Controller projects lifecycle and Brain decides any degraded-state action
Example Payload Shapes

ACTIVE IntentSpecification Used by Runtime Intent

The ACTIVE IntentSpecification is the governed contract. The runtime Intent triggers fulfilment by referencing that ACTIVE specification ID and supplying targets, constraints and preferences.

ACTIVE IntentSpecification
{
  "id": "ispec-hss-001",
  "version": "1.1",
  "lifecycleStatus": "ACTIVE",
  "expressionSpecification": {
    "iri": "https://mycsp.com.au/tio/hospital-surgical-slice/v1.0"
  },
  "@type": "IntentSpecification"
}
Runtime Intent using the ACTIVE spec
{
  "name": "Sydney Hospital Surgical Connection Intent",
  "intentSpecification": {
    "id": "ispec-hss-001"
  },
  "expression": {
    "iri": "https://mycsp.com.au/tio/hospital-surgical-slice/v1.0",
    "expressionValue": {
      "context": {
        "targets": {
          "minAvailabilityPercent": 99.99
        },
        "constraints": {
          "serviceClass": "critical-gold",
          "locationId": "AU-NSW-SYD-HOSP-001"
        },
        "preferences": {
          "preferredAccessTechnology": "5G"
        }
      }
    }
  },
  "@type": "Intent"
}
Key distinction

Only an ACTIVE IntentSpecification can be used for new runtime Intent admission. IC MS validates the runtime request against the referenced specification before the intent moves into downstream resolution and execution. The example omits submit because the normal path is immediate submission; submit=false is only for draft authoring.

Deep Dive · Microservice 1

ID MS (Intent Definition) — Librarian for IntentSpecifications

Definition-time catalogue and version governance

📚
Role
Librarian (ID MS) owns the governed IntentSpecification catalogue that defines which runtime Intent requests are allowed.
TMF-aligned REST surface
Create/List POST, GET /intentManagement/v5/intentSpecification
Retrieve/Update GET, PATCH /intentManagement/v5/intentSpecification/{id}
Retire DELETE /intentManagement/v5/intentSpecification/{id}
Subscribe POST /intentManagement/v5/intentSpecification/hub
Key responsibilities
  • Creates and governs IntentSpecification drafts and active versions.
  • Maintains version lifecycle: DRAFT, ACTIVE and RETIRED.
  • Protects governed changes with optimistic concurrency using ETag and If-Match.
Boundary: does not own
  • Does not admit runtime Intents.
  • Does not resolve intent meaning or select network resources.
  • Does not rewrite historical runtime context when a specification is retired.
Why it matters
Keeps runtime admission safe by making the allowed contract explicit, versioned and governed.

Main outputs

Deep Dive · Microservice 3

II MS (Intent Intelligence) — Brain For Intents

Meaning, decision coordination and service-ready configuration

🧠
Role
Brain (II MS) interprets intent meaning, uses KP context, coordinates approved decision systems and resolves service-ready configuration.
Key responsibilities
  • Consumes admitted IntentValidatedEvent workflow input.
  • Uses KP context: policy, topology models, benchmarks, constraints, freshness rules and trust signals.
  • Coordinates approved systems such as Inventory, CRM, identity, policy, optimiser and domain sources where needed.
  • Produces IntentNetworkReadyEvent when configuration is ready for apply and monitoring.
  • Consumes IntentAssuranceEvent when assurance indicates degraded-state decisioning may be needed.
Boundary: does not own
  • Does not own external system-of-record data.
  • Does not apply configuration directly.
  • Does not project external runtime lifecycle state.
Why it matters
Turns an admitted customer intent into an actionable, governed configuration decision.
Main inputs
IntentValidatedEvent KP governed context Decision outcomes IntentAssuranceEvent for governed control-loop input
Main outputs
IntentNetworkReadyEvent IntentRejectedEvent Governed degraded-state decisions

Infrastructure Pattern

Event Architecture & Kafka Topology

3 Delivery Models — Must Not Be Collapsed
① Internal Kafka Events
CloudEvents-style headers + plain JSON body. At-least-once delivery. Idempotent consumers required. Used for all internal workflow facts.
② External Webhooks
HTTP POST to subscriber callback URLs. Service-owned outbox + retry relay. TMF-aligned payloads. Used by ID MS and IC MS for subscriber notifications.
③ Callback Ingestion
REST POST to ICB MS. Outbox persistence before Kafka relay. Two profiles: change-execution callbacks go to IA MS; optimisation status callbacks go to II MS.

Security Architecture

Defence-in-Depth Security Model

🔐
User Authentication
OEX users authenticated via identity provider. Authorised sessions only. All requests carry verified identity through API Gateway.
🚪
API Gateway (NGW)
ID MS, IC MS, ICB MS exposed through NGW. NGW authenticates and forwards trusted identity claims. External callers cannot reach services directly.
🛡️
Service-to-Service
Internal traffic uses mTLS or equivalent. II MS and IA MS are internal-only — not exposed via NGW or any external-facing interface.
📨
Kafka Security
Topic-level ACLs. Producers and consumers authenticated. Event integrity enforced. Sensitive data (secrets, tokens) never included in event payloads.
🔁
Callback Trust Boundary
ICB MS performs technical authorisation — validates caller identity from NGW-forwarded claims before accepting any callback submission. No blind trust.
🔒
API Concurrency
Optimistic locking or equivalent for concurrent updates. Idempotency keys on write operations. ETag or version headers for resource-level conflict detection.

Operationalisation

Delivery and Operational Readiness

How the platform is built, released, scaled, observed and recovered in production.

Elastic deployment and cost control

Each microservice owns its persistence, cache strategy and Helm-based deployment unit. Autoscaling uses CPU, memory, request throughput, error rate, latency and queue-depth signals so the platform can scale under load while avoiding unnecessary infrastructure cost.

Regional resilience posture

Platform services, persistence, eventing, external endpoints and SaaS choices are selected so the deployment can evolve from single-region to active-passive regional failover without redesigning the platform.

CI/CD and release governance

Each microservice should have an independent pipeline for build, test, security scan, contract validation, container image promotion and Helm-based deployment. Pipelines should support environment promotion, rollback, configuration control and approval gates for production changes.

Automated quality validation

Automated functional, load and performance testing should be combined with chaos-engineering practices to prove the platform meets defined quality attributes before production rollout and during controlled release cycles.

Observability, traceability and visualisation

Centralised logs, metrics, traces, events, dashboards and alerts. Candidate stack: Prometheus, Grafana, Jaeger, Kiali and service mesh. Views cover request health, event lag, callback failures, latency, saturation and end-to-end traceability.

Operational readiness and runbooks

Production readiness should include runbooks for DLQ handling, outbox recovery, callback failures, dependency timeouts, regional failover, rollback and controlled manual intervention when the use-case autonomy boundary requires approval.

Architecture Principles and Recommended Next Steps

The decisions that should remain stable as the platform moves from architecture into delivery.

Clear ownership

Librarian, Controller, Brain, Dispatcher and Guardian each own a distinct platform responsibility.

Governed admission

Only ACTIVE IntentSpecifications admit new runtime Intents, with version and lifecycle governance.

External execution ownership

Intent Management derives, hands over and assures configuration; the network or change-execution layer owns execution state.

Feedback-loop architecture

External systems(Tx) send status back through Dispatcher, Guardian evaluates assurance, and Brain decides degraded-state control-loop action.

Future regional resilience

Platform services and supporting technologies are selected so active-passive or regional failover remains possible later.

Autonomy boundary

Each use case defines what can run automatically, what needs approval, and what remains manual.

TMF autonomy positioning
Autonomy position
How this platform supports it
L3-style assisted closed loop
Detect assurance variance, surface IntentReport evidence, and support governed operator-approved action where the use case requires approval.

Positioning is use-case bounded. This is not a claim of unrestricted full autonomy across every domain from day one.
